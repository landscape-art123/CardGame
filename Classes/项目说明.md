# 纸牌匹配 Demo 项目说明

## 1. 项目背景

这个项目是我基于 **C++ + cocos2d-x 3.17** 实现的一个纸牌匹配 Demo，主要目的是按照给定需求完成一个小型休闲玩法，并在实现过程中练习客户端开发里比较重要的几件事：

- 界面与交互实现
- 状态管理
- Undo 回退设计
- 工程化分层
- 面向后续扩展的代码组织

这次需求本身聚焦在三项核心功能：

1. 点击手牌后，手牌移动到顶部牌位置并替换顶部牌。
2. 点击桌面牌后，如果与顶部牌点数差 1，则移动到顶部牌位置并替换顶部牌。
3. 支持连续回退，能够按操作顺序的反方向恢复牌的位置与顶部牌状态。

因此，这个项目的重点不是做完整纸牌游戏，而是把一个基础玩法从 **界面、交互、逻辑到结构设计** 做完整。

---

## 2. 需求理解

根据设计文档，本次实现的目标比较明确：

- **需求 1：手牌区翻牌替换**
  点击手牌区中的牌，牌面通过 `MoveTo` 动画移动到顶部牌位置，并替换顶部牌。

- **需求 2：桌面牌与顶部牌匹配**
  桌面牌只要与顶部牌点数差 1 即可匹配，无花色要求；匹配成功后，桌面牌移动到顶部牌位置并替换顶部牌。

- **需求 3：回退功能**
  支持连续点击回退按钮，将最近一次操作按相反顺序恢复，直到没有回退记录为止。

除了功能点，文档还明确提出了两类要求：

1. 界面按指定分辨率与区域尺寸实现。
2. 代码结构尽量遵循 **MVC**，保证后续的可维护性和可扩展性。

这说明这个题目除了考察基本功能实现，还在考察工程化思路。

---

## 3. 开发环境

- 语言：C++
- 引擎：cocos2d-x 3.17
- 设计分辨率：1080 × 2080
- 主牌区尺寸：1080 × 1500
- 卡堆区尺寸：1080 × 580

在实现时，我先把界面比例、卡牌大小、上下区域位置关系对齐，再逐步整理代码结构，避免一边重构一边调布局导致整体混乱。

---

## 4. 工程化实现思路

虽然这个项目规模不大，但我没有把所有逻辑直接堆到一个 Scene 文件里，而是尽量按小型客户端项目的方式拆分职责。

我主要关注的是下面几个问题：

- 哪些内容属于运行时状态
- 哪些内容属于界面展示
- 哪些内容属于交互和业务逻辑
- 哪些内容可以抽成通用服务
- Undo 如何做成可扩展机制

基于这些考虑，项目最终按 **MVC + Manager + Service** 的思路做了拆分。

---

## 5. MVC 分层设计

### 5.1 Model

Model 层只负责保存运行时数据，不处理复杂业务逻辑。

主要包括：

- `CardModel`：保存单张牌的点数、花色、位置、是否已移除等信息
- `UndoModel`：保存一次操作对应的回退数据
- `GameModel`：统一保存当前整局状态，包括顶部牌、手牌、左侧牌和桌面牌列表

这样做的好处是，状态集中、逻辑清晰，后续如果继续加玩法或加字段，不需要到处找散落的成员变量。

### 5.2 View

View 层只负责界面展示、输入监听和动画表现，不负责业务判断。

主要包括：

- `CardView`：负责单张牌的显示
- `GameView`：负责整体界面创建、区域节点、卡牌刷新、点击监听和 MoveTo 动画

我把牌面显示、按钮区域、点击监听、动画表现都放到 View 层，这样 View 只关心“怎么显示”和“用户点了哪里”，不关心“这一点是否有效”。

### 5.3 Controller

Controller 层负责连接 View 和 Model，处理用户操作对应的业务逻辑。

主要拆成了三块：

- `GameController`：负责整体流程、初始化、Undo 调度
- `PlayFieldController`：负责桌面牌点击后的匹配逻辑
- `StackController`：负责手牌点击后的替换逻辑

这样做以后，职责会比较清楚：

- 点桌面牌，进入 `PlayFieldController`
- 点手牌，进入 `StackController`
- Undo 由 `GameController` 统一处理

### 5.4 Manager

Manager 层主要负责可复用的流程管理。

当前项目里最典型的是：

- `UndoManager`

它只负责管理回退记录的入栈和出栈，不直接依赖具体界面，也不负责牌面规则判断。这样以后如果新增“抽牌回退”或“翻牌回退”，还能继续复用同一套 Undo 机制。

### 5.5 Service

Service 层主要放无状态的业务逻辑。

当前主要包括：

- `CardRuleService`：判断两张牌是否满足点数差 1 的匹配条件
- `GameModelFromLevelGenerator`：根据固定布局生成一局运行时 `GameModel`

这类逻辑不直接操作场景节点，也不持有界面状态，更适合做成独立服务。

## 6. 当前交互流程

### 6.1 点击桌面牌

1. `GameView` 监听到桌面牌点击。
2. 将 `cardId` 回调给 `PlayFieldController`。
3. `PlayFieldController` 调用 `CardRuleService` 判断当前牌是否与顶部牌差 1。
4. 如果可匹配，则先记录一条 `UndoModel`。
5. 更新 `GameModel` 中的桌面牌状态与顶部牌状态。
6. 调用 `GameView` 播放桌面牌移动到顶部牌位置的动画，并刷新顶部牌显示。

### 6.2 点击手牌

1. `GameView` 监听到手牌点击。
2. 回调给 `StackController`。
3. `StackController` 先记录 Undo。
4. 更新 `GameModel` 中的顶部牌状态。
5. 调用 `GameView` 播放手牌移动到顶部牌位置的动画，并刷新顶部牌。

### 6.3 点击回退

1. `GameView` 监听到“回退”点击。
2. 回调给 `GameController`。
3. `GameController` 从 `UndoManager` 中弹出最近一条记录。
4. 先恢复顶部牌。
5. 再根据来源类型恢复桌面牌或手牌的位置。
6. 刷新界面与运行时状态。

这样以后，界面、逻辑、状态三部分之间的关系会比较清楚，后续加功能也不会全部挤在同一个类里。

## 7. 随机生成的实现

在基础需求完成以后，我又额外考虑了一个更接近实际游戏开发的问题：

如果不想把牌面全部写死，怎么生成一局数据？

### 7.1 第一版思路

我目前采用的是：

- **位置固定**
- **牌面随机**

也就是说：

- 主牌区和底部区域的位置由布局配置固定
- 牌的点数和花色由生成器随机生成

这样做的好处是：

1. 不会破坏当前已经调好的界面布局
2. 容易接入现有 MVC 结构
3. 便于从“写死 demo 数据”过渡到“生成一局可运行数据”

### 7.2 当前约束

为了避免完全随机后出现“开局一张都点不了”的情况，我给第一版随机生成加了一个简单约束：

- 保证开局至少有一张桌面牌能和顶部牌匹配

这样既保留了随机性，也保证了最基本的可玩性。

---

## 8. 后续保证通关的设想

我现在已经考虑了随机生成，但还没有把整局一定保证通关完整做完。

如果后续继续做，我计划分两步：

### 8.1 先做生成后检测

思路是：

1. 先随机生成一局
2. 检查当前局面是否有解
3. 如果无解，则重新生成

这种方式实现难度相对可控，适合作为下一阶段的扩展。

### 8.2 再做逆向构造

更进一步的话，可以从最终可完成状态反向构造初始局面，这样生成出来的局天然存在一条解法路径。

这个思路更适合真正的关卡生成系统，也更接近休闲 puzzle 类游戏里对关卡可玩性的要求。

---

## 9. 可扩展性考虑

这次项目我比较在意的一点，是后续继续加东西时，尽量不要把原有结构改乱。

### 9.1 新增卡牌类型

如果以后要新增牌型或扩展牌面属性，主要修改点会集中在：

- `CardModel`
- `CardView`
- `GameModelFromLevelGenerator`
- `CardRuleService`

因为数据、显示和规则已经拆开，所以不会影响太多现有流程代码。

### 9.2 新增回退类型

如果后续还要支持：

- 抽牌回退
- 翻牌回退
- 发牌回退

主要扩展位置会是：

- `UndoModel`
- `UndoManager`
- 对应 Controller 中的记录与恢复逻辑

这样 Undo 机制还是统一的，不会变成每个功能都单独写一套。

## 10. 和岗位的匹配理解

这个项目虽然只是一个小型纸牌 Demo，但我在实现时，主要是按**游戏客户端开发**的思路去做，而不是只把需求硬写出来。

从岗位方向看，客户端工作更偏：

- C++ / cocos 开发
- 游戏内功能与组件实现
- 和产品、策划或工具平台一起推进需求落地
- 持续迭代玩法和功能模块

所以我在这个项目里重点练习的，其实不是复杂算法，而是：

### 10.1 需求拆解能力

把一个小玩法拆成：

- 运行时状态
- 界面表现
- 业务逻辑
- 通用服务

### 10.2 工程化代码组织

例如：

- 目录结构是否清楚
- 模块职责是否单一
- 后续扩展会不会很痛苦
- 是否能避免把所有逻辑都堆进一个 Scene

### 10.3 面向迭代的写法

当前需求只要求点击匹配和 Undo，但我已经把：

- 随机生成
- Undo 扩展
- 可解性检测
- 保证通关

这些方向提前考虑到了结构位置。

我觉得这会更接近真实项目里的工作方式：
不是一次把所有功能写死，而是先完成当前版本，再为后续迭代留下空间。

---

## 11. 总结

这个纸牌匹配 Demo 对我来说，不只是一次功能实现练习，更像是一次小型客户端工程训练。

我在这个项目里完成了：

- cocos2d-x 下的基础界面与交互实现
- 手牌替换、桌面牌匹配、连续 Undo 的完整链路
- 基于 MVC 的结构拆分
- 运行时状态的统一管理
- 随机生成的第一版设计
- 后续保证通关的扩展思路预留

我觉得这个项目最大的价值是能体现出面对一个小需求时，尽量按**可维护、可扩展、接近真实客户端开发**的方式去实现，而不是只把 demo 拼出来。
